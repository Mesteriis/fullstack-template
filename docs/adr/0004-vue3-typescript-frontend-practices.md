# ADR-0004: Практики Vue 3 + TypeScript frontend

- Статус: Accepted
- Дата: 2026-03-25

## Контекст

Монорепозиторий должен поддерживать web-приложение без превращения корня frontend-приложения в плоский каталог из компонентов, API-хелперов и случайных composable. Нужны правила, которые дадут предсказуемую структуру и совместимость со spec-first подходом.

## Решение

Для deployable frontend-приложения принимается следующая структура:

```text
src/frontend/
  package.json
  package-lock.json
  vite.config.ts
  index.html
  main.ts
  App.vue
  app/                 # bootstrap, router, providers, layouts
  pages/               # route-level экраны
  features/            # бизнес-фичи
  entities/            # доменные сущности и повторно используемые блоки
  shared/
    api/               # generated client adapters, query helpers
    ui/                # базовые UI-компоненты
    lib/               # утилиты без бизнес-логики
    config/            # frontend-конфигурация
    assets/            # статические ресурсы
  tests/
    unit/
    component/
```

Правила разработки:

- Используется Vue 3 + TypeScript. Основной стиль компонентов: Composition API и `script setup`. Options API допускается только как исключение для совместимости со сторонними библиотеками.
- Frontend живёт в `src/frontend/`, а его toolchain manifests (`package.json`, lock-файл, `vite.config.ts`) лежат рядом с приложением, а не в корне репозитория.
- В минимальной стадии допустимы `main.ts`, `App.vue` и базовые конфиги в корне `src/frontend/`. По мере роста приложение раскладывается по `app/`, `pages/`, `features/`, `entities/` и `shared/`.
- TypeScript работает в строгом режиме. `any` в коде приложения запрещается, кроме локально обоснованных адаптеров на границе со сторонним кодом.
- Компонент по умолчанию живёт рядом с фичей или страницей. Перенос в `shared/ui` допускается только после подтверждённого повторного использования.
- Компоненты и страницы не ходят в API напрямую. Работа с сетью инкапсулируется в `shared/api` и feature-level composable.
- Глобальное состояние через Pinia используется только для сессии, авторизации и действительно shared-state между маршрутами. Локальное UI-состояние не выносится в store.
- DTO и API-типы берутся из generated artifacts, полученных из спецификаций. Ручное дублирование типов рядом с компонентами запрещается.
- В `pages/` размещается композиция фич и сценарии маршрутов, а не низкоуровневая переиспользуемая логика.
- Базовые визуальные примитивы и токены находятся в `shared/ui`; если они начинают использоваться несколькими приложениями, они выносятся в `packages/frontend/`.

Правила тестирования:

- unit/component тесты выполняются через Vitest и живут в `src/frontend/tests`;
- пользовательские сценарии на уровне приложения выполняются через `tests/e2e` репозитория;
- критичные формы и интеграционные точки проверяются через тесты с моками generated client или contract fixtures.

## Последствия

Положительные:

- структура frontend-приложения остаётся читаемой по мере роста;
- frontend естественно интегрируется со spec-first и generated clients;
- локальная UI-логика не разрастается в глобальное состояние без причины.

Отрицательные:

- команде придётся сознательно поддерживать границы между `pages`, `features`, `entities` и `shared`;
- часть простых экранов будет требовать чуть более строгой раскладки по каталогам, чем в ad-hoc подходе.
