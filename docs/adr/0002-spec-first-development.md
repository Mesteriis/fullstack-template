# ADR-0002: Spec-First разработка

- Статус: Accepted
- Дата: 2026-03-25

## Контекст

Проект будет развиваться как минимум в двух плоскостях: backend и frontend. Если контракт API описывается постфактум, frontend начинает дублировать DTO вручную, backend публикует неописанные поля, а интеграционные ошибки обнаруживаются слишком поздно.

## Решение

Исходным источником истины для внешних контрактов считаются спецификации в каталоге `specs/`, а не реализация в коде.

Распределение по типам контрактов:

- `specs/openapi/<service>/openapi.yaml` для синхронных HTTP API;
- `specs/asyncapi/<service>/asyncapi.yaml` для событий, очередей и брокеров;
- `specs/jsonschema/<domain>/` для общих схем данных, используемых несколькими приложениями.

Обязательный порядок изменений:

1. Изменяется спецификация.
2. Обновляются примеры и версии контракта.
3. Перегенерируются клиенты, типы и серверные заготовки в `packages/generated/`.
4. Реализуется backend/frontend код.
5. Добавляются или обновляются контрактные тесты.

Дополнительные правила:

- Backend не публикует новые endpoint, поля или события без соответствующего изменения спецификации.
- Frontend не описывает API DTO вручную, если эти типы уже могут быть получены из спецификации.
- Любой generated code хранится отдельно от handwritten code и не редактируется вручную.
- Любое breaking change требует явного повышения версии контракта и краткой migration note рядом со спецификацией.
- Для каждой спецификации должны храниться примеры успешных и ошибочных сценариев.
- CI должен проверять валидность спецификаций, отсутствие дрейфа generated artifacts и прохождение контрактных тестов.

## Последствия

Положительные:

- backend и frontend синхронизируются через общий контракт;
- снижается число интеграционных регрессий и ручного дублирования типов;
- спецификация становится ревьюируемым артефактом, а не побочным продуктом реализации.

Отрицательные:

- любое API-изменение требует дополнительного шага со спецификацией и генерацией;
- команде нужно дисциплинированно поддерживать версионирование и примеры контрактов.
