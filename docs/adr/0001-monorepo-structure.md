# ADR-0001: Целевая структура монорепозитория

- Статус: Accepted
- Дата: 2026-03-25

## Контекст

Репозиторий уже содержит backend-заготовку, тесты и Docker-окружение, но пока не фиксирует, как в нём должны сосуществовать backend, frontend, контракты, общие библиотеки и инфраструктурные артефакты. Без явной структуры монорепозиторий быстро превращается в набор каталогов без понятных границ ответственности.

## Решение

Принимается следующая целевая структура репозитория:

```text
src/
  backend/              # deployable Python/FastAPI backend
  frontend/             # deployable Vue 3 + TypeScript frontend
packages/
  python/               # общие Python-библиотеки
  frontend/             # общие frontend-пакеты и UI-kit
  generated/            # только сгенерированный код
specs/
  openapi/              # HTTP API контракты
  asyncapi/             # события и брокеры сообщений
  jsonschema/           # общие схемы данных
docs/
  adr/                  # архитектурные решения
alembic/                # корневые миграции БД через Alembic
alembic.ini             # конфигурация Alembic
docker/                 # Dockerfile, compose, entrypoint-скрипты
infra/                  # IaC, deployment manifests, окружения
scripts/                # служебные скрипты и automation
tests/
  contract/             # межсервисные контрактные проверки
  e2e/                  # end-to-end сценарии через несколько приложений
```

Правила размещения:

- `src/` является нейтральным корнем для исходного кода приложений. Это polyglot-каталог, а не Python-специфичный namespace.
- `src/backend/` и `src/frontend/` содержат только код и локальные артефакты соответствующего приложения: манифесты зависимостей, `tests/`, `README.md`, конфигурацию сборки и запуска.
- `packages/*` содержат только переиспользуемый код. Если модуль нельзя независимо переиспользовать, он остаётся внутри конкретного `app`.
- `packages/generated/*` создаются из спецификаций или иных генераторов. Ручные правки в этих каталогах запрещены.
- `specs/*` являются домом для контрактов, примеров запросов/ответов и описаний версий контрактов.
- Unit, integration и component тесты живут рядом с приложением в `src/backend/tests` и `src/frontend/tests`. Корневой `tests/` используется только для cross-app сценариев: `contract` и `e2e`.
- Документация по архитектуре и соглашениям хранится в `docs/`; ADR всегда хранятся в `docs/adr`.
- Миграции основной PostgreSQL-базы хранятся в корневом `alembic/`; для них используется Alembic и корневой `alembic.ini`.
- Языковые манифесты и lock-файлы лежат рядом с owning app, а не в корне репозитория. Примеры: `src/backend/pyproject.toml`, `src/backend/uv.lock`, `src/frontend/package.json`, `src/frontend/package-lock.json`.
- Корневые манифесты допустимы только для repo-wide orchestration и не должны делать монорепу визуально "владением" одного стека. Типовые примеры: `.gitignore`, `README.md`, `docker-compose.yml`, `pnpm-workspace.yaml`, `turbo.json`, `justfile`, `Makefile`.
- `docker/` используется для контейнеризации и локальной сборки, `infra/` для окружений и deployment-артефактов. Эти роли не смешиваются.

## Последствия

Положительные:

- становится понятно, где искать приложение, общий код, контракты и скрипты;
- frontend и backend могут развиваться в одном репозитории без смешивания ответственности и без доминирования Python-артефактов в корне;
- появляется предсказуемое место для сгенерированного кода и контрактных тестов.
- корень репозитория остаётся нейтральным по отношению к стеку и лучше масштабируется под новые языки и инструменты.

Отрицательные:

- команды запуска становятся более адресными, потому что backend и frontend имеют собственные локальные манифесты;
- при добавлении новых модулей нужно соблюдать границы `src`, `packages` и корневых repo-level артефактов, а не складывать всё в корень проекта.
