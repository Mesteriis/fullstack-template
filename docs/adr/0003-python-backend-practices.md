# ADR-0003: Практики Python backend

- Статус: Accepted
- Дата: 2026-03-25

## Контекст

В репозитории уже есть Python/FastAPI backend. Чтобы backend не деградировал в набор роутов с бизнес-логикой и случайных интеграций, нужны явные правила по структуре модулей, типизации, конфигурации и тестам.

## Решение

Для deployable backend-приложения принимается следующая структура:

```text
apps/api/
  <service_name>/
    api/               # FastAPI routers, dependencies, transport schemas
    application/       # use cases, orchestration, ports
    domain/            # бизнес-сущности и чистые правила
    infrastructure/    # async SQLAlchemy/PostgreSQL, HTTP clients, adapters, repositories
    settings/          # конфигурация приложения
    main.py            # сборка FastAPI app и wiring
  tests/
    unit/
    integration/
alembic/
alembic.ini
```

Правила разработки:

- `uv` используется как стандартный инструмент для управления зависимостями и запуска команд.
- Все публичные функции, методы и модели имеют явную типизацию. Непротипизированный production-код не считается допустимым по умолчанию.
- `ruff` используется для lint/format, `pytest` для тестов, статическая проверка типов обязательна через `mypy` или `pyright`.
- Каталог `src/` не используется. Пакет приложения живёт напрямую внутри `apps/api/`.
- FastAPI и Pydantic ограничиваются transport/boundary-слоем. Доменные сущности и application-логика не должны зависеть от FastAPI.
- Бизнес-логика не размещается в роутерах. Роутер принимает запрос, валидирует вход и делегирует use case.
- Основной стек backend: FastAPI, Alembic, async PostgreSQL.
- Работа с PostgreSQL идёт асинхронно; доступ к БД и репозитории находятся в `infrastructure/`. Рекомендуемый базовый стек для этого слоя: SQLAlchemy asyncio + asyncpg.
- Миграции БД хранятся в корневом `alembic/` и управляются через Alembic, а не через самодельные SQL-скрипты.
- Конфигурация читается централизованно через `settings/`; прямое чтение переменных окружения по всему коду запрещается.
- Для I/O используется `async`, но блокирующие операции не допускаются внутри async-пути без явного вынесения.
- API-схемы, доменные модели и persistence-модели не смешиваются в один класс "на все случаи".

Правила тестирования:

- unit-тесты покрывают `domain/` и `application/` без сети и без реальной БД;
- integration-тесты проверяют адаптеры, БД и внешние клиенты на уровне приложения;
- contract-тесты живут на уровне репозитория в `tests/contract`, если они проверяют соответствие OpenAPI/AsyncAPI.

## Последствия

Положительные:

- backend остаётся модульным и тестопригодным;
- бизнес-правила можно проверять без FastAPI и инфраструктуры;
- становится проще заменять БД, клиентов и транспортный слой.

Отрицательные:

- появляется больше явных слоёв и файлов;
- потребуется дисциплина, чтобы не протаскивать FastAPI/Pydantic/ORM в domain-слой.
